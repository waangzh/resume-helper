# Resume Tailor

## 流程

1. 读取岗位画像、候选人画像和经历筛选结果。
2. **必须读取** `references/resume-standards.md`，严格遵守其中的 Markdown 输出格式规范。
3. 根据 `JobProfile.role_category` 选择模板结构：
   - 技术岗：基本信息 → 教育背景 → 技术技能 → 项目经历 → 实习经历 → 竞赛奖项
   - 通用岗：基本信息 → 教育背景 → 实习经历 → 校园经历 → 社会实践 → 竞赛奖项 → 技能与证书
4. 先压缩内容再润色：默认一页优先，保留最能证明目标岗位匹配度的内容。
5. 使用 STAR 原则改写经历 bullet，每条尽量 1-2 行。
6. 对缺少量化结果的内容使用保守表达；无法从资料确认的事实只进入待确认项，不写入最终简历正文。
7. 输出可交给 `$resume-export` 的 `TailoredResume`，待确认项放入 `pending_confirmations` 字段，**不在简历正文或 HTML 中渲染**，只在交付摘要中展示。

## Markdown 输出规范（核心约束）

> 格式错误会导致 HTML 预览排版混乱，必须严格遵守。

**分隔符约定：**
- 基本信息、教育背景字段间：使用 **两个以上空格** 或 **Tab**
- 求职意向：只显示岗位，格式为 `求职意向：{岗位}`
- 竞赛奖项字段间：使用 **两个以上空格**

**关键格式要点：**
- 一级标题只写姓名
- 教育背景必须三行完整：学校专业时间行、成绩行、课程行
- 项目标题格式：`### {项目名}  {时间}`（两个以上空格分隔）
- 实习标题格式：`### {公司} / {部门} / {岗位}  {时间}`
- 校园经历标题格式：`### {组织} / {职务}  {时间}`
- 每个 bullet 以 `- ` 开头（短横线 + 空格）

详见 `references/resume-standards.md` 的完整格式规范。

## 改写规则

### 技术岗

- 使用招聘方能识别的技术关键词，但必须有经历证据。
- 避免空泛词：参与、负责部分工作、学习了、了解了。
- 不凭空加入 Redis、性能优化、并发、上线部署等能力。
- 如果用户材料较弱，给出”可补充方向”，不要写成既成事实。

### 通用岗

- 使用 STAR 原则描述经历，突出职责和成果。
- 避免空泛词：参与了、协助了、帮忙了。
- 突出软技能证明：沟通、协作、组织、执行力。
- 兼职/社团经历需有具体数字或成果。

### 通用规则

- 默认一页优先，应届/实习简历控制在 1 页；内容多时删减低相关信息，不拉长流水账。
- 使用清晰标题、标准模块和纯文本友好结构，避免复杂表格、花哨图标、进度条等影响 ATS 或编辑的元素。

## 数量边界

- 经历数量以用户提供的材料为准，**不自动补充**
- 只有用户主动提出要求时，才可从其他材料中提取或建议额外经历
- 用户只提供 1 段经历，就只输出 1 段经历
