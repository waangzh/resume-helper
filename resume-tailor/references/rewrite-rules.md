# 简历改写规则

## STAR 原则改写指南

> STAR 是经历描述的标准框架，适用于项目、实习、社团、兼职等各类经历的职业化改写。

### STAR 四要素

| 要素 | 含义 | 关键问题 | 示例 |
|------|------|----------|------|
| **S**ituation | 情景背景 | 在什么场景下？团队/组织规模？ | 学生会外联部干事（20人团队） |
| **T**ask | 任务目标 | 要解决什么问题？承担什么职责？ | 负责校企合作活动的赞助招商 |
| **A**ction | 具体行动 | 做了什么？用了什么方法？ | 筛选目标企业50家，电话沟通30家，实地拜访8家 |
| **R**esult | 可量化结果 | 成果是什么？有什么影响？ | 成功拉取赞助8000元，覆盖活动成本60% |

### STAR 改写模板

#### 模板一：完整 STAR（推荐）

```markdown
- {情景}：{背景简述}。{任务}：{目标或职责}。{行动}：{具体动作}。{结果}：{量化成果}
```

**示例：**
```markdown
- 在学生会外联部（20人团队）负责校企合作活动赞助招商，筛选目标企业50家，电话沟通30家，实地拜访8家，成功拉取赞助8000元，覆盖活动成本60%
```

#### 模板二：精简版（空间有限时）

```markdown
- {角色/职责}，{核心行动}，{量化结果}
```

**示例：**
```markdown
- 负责校企合作赞助招商，拜访企业8家，拉取赞助8000元
```

### STAR 自检清单

| 检查项 | 是否通过 |
|--------|----------|
| 是否有清晰的情景背景（S）？ | ☐ |
| 是否明确了任务或职责（T）？ | ☐ |
| 是否描述了具体行动而非泛泛而谈（A）？ | ☐ |
| 是否有可量化的结果（R）？ | ☐ |
| 结果是否真实可信、可追溯？ | ☐ |
| 整体是否控制在1-2行内？ | ☐ |

### 无量化结果时的处理

当无法提供数字时，用具体交付物替代：

| 有数据 | 无数据时的替代 |
|--------|----------------|
| 拉取赞助8000元 | 成功拉取赞助，覆盖活动主要支出 |
| 粉丝增长1300人 | 粉丝数增长6倍 |
| 获省赛二等奖 | 获省级竞赛奖项 |
| 提升至85分 | 成绩显著提升，家长主动续课 |

## Bullet 句式模板

> 使用”动作 + 技术/任务 + 结果”结构，每条控制在 1-2 行。

### 推荐句式

| 句式 | 示例 |
|------|------|
| 基于 {技术} 实现 {功能}，用于 {场景} | 基于 Redis 实现热点数据缓存，用于商品详情页加速 |
| 设计并完成 {产出}，提升 {指标} | 设计并完成订单状态机，提升订单处理可维护性 |
| 针对 {问题}，通过 {方案} 完成 {结果} | 针对慢查询问题，通过索引优化将响应时间从 2s 降至 200ms |
| 在 {项目} 中承担 {职责}，交付 {成果} | 在电商项目中承担后端开发，交付商品管理模块 |

### 禁止句式

| 禁止 | 原因 |
|------|------|
| 精通/深入掌握 | 除非资料中有强证据 |
| 大幅提升/显著优化 | 必须有指标或具体动作 |
| 独立负责全部系统 | 若资料只显示参与 |
| 熟练使用/精通 | 若只有课程学习或无项目证据 |
| 提升 XX% | 若没有来源、口径或待确认说明 |

## 待确认写法

当缺少关键信息时，使用以下格式标注：

```markdown
可补充：接口数量、数据规模、性能指标或上线情况
待确认：该项目中是否使用 Redis 缓存或消息队列
```

## Bullet 自检清单

| 检查项 | 是否通过 |
|--------|----------|
| 是否说明个人动作，而不是团队整体成果 | ☐ |
| 是否包含岗位关键词，但不过度堆砌 | ☐ |
| 是否能被面试追问时讲清楚 | ☐ |
| 是否有数字、范围、结果；没有则用具体交付物替代 | ☐ |
| 是否符合 Markdown 格式规范（`- ` 开头） | ☐ |

## 量化表达指南

| 有数据时 | 无数据时 |
|----------|----------|
| 接口响应时间降低 60% | 完成 XX 模块的核心接口开发 |
| 核心模块覆盖率达 85% | 编写单元测试 50+ 用例 |
| 支持 1000+ 并发用户 | 支持多用户并发访问 |
| 日均处理 10 万条数据 | 处理业务数据并生成报表 |
