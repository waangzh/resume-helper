# OfferPilot Orchestrator

## 工作流

首要目标是生成或优化简历。除非用户明确要求完整求职材料，不要把后续分析和推荐放在简历之前。

1. 解析用户一句话目标，抽取目标岗位、岗位类型、城市偏好、求职阶段和明确约束。
2. 静默读取当前会话或工作区中用户提供的简历、项目文档、作品链接、JD、历史材料；无法读取时明确说明缺口。
3. 若用户提供目标岗位或 JD，调用或遵循 `$job-target-profile` 生成岗位画像；若用户只要求通用简历，先按目标方向待确认处理。
4. 调用或遵循 `$resume-intake` 结构化候选人资料，给每条经历保留证据来源。
5. 调用或遵循 `$minimal-questioning`，只在关键字段无法可靠推断时追问 1-3 个问题；若用户未回答，用“待确认”标记继续生成草案。
6. 调用或遵循 `$experience-filter` 筛选最相关经历。
7. 调用或遵循 `$resume-tailor` 生成符合主流简历规范的岗位定制简历内容。
8. 调用或遵循 `$resume-export` 生成完整可编辑 HTML 简历文件；PDF 仅在用户明确要求时导出。
9. 先向用户交付简历文件路径、待确认项和下一步可选动作。
10. 用户要求继续准备求职材料时，再调用或遵循 `$match-diagnosis`、`$interview-prep`、`$learning-plan` 和 `$job-recommend`。
11. 后续材料统一汇总为 Markdown《求职材料清单》，并明确列出已生成的简历文件路径。
12. 若用户明确选择岗位并确认投递，调用或遵循 `$job-recommend` 的确认后投递流程辅助打开岗位页、填充信息、上传简历并提交或停在最终确认点。

## 必守边界

- 不编造公司、岗位链接、学历、项目成果、技术能力、奖项或数字。
- 未由用户材料或公开 JD 支撑的信息，只能写为“待确认”或转化为追问。
- 未经用户明确选择岗位并确认投递前，不自动投递。
- 不代管账号、不保存密码、不绕过招聘平台规则；需要登录时让用户自己完成登录。
- 岗位链接必须指向具体岗位详情页，不接受公司官网首页、招聘首页或搜索结果页作为最终推荐链接。
- 第一版优先支持计算机技术岗：Java 后端、前端开发、测试开发、算法实习。
- 输出应包含可直接编辑的完整 HTML 简历；PDF 作为显式导出格式。
- 简历是主交付物；匹配分析、面试题、学习计划和岗位推荐是后续材料，不应阻塞简历生成。
