# 诊断 Rubric

## 诊断分类

| 类别 | 判断标准 | 下一步动作 |
|------|----------|------------|
| 已命中能力 | 岗位核心技能在简历项目、实习或技能栈中有明确证据 | 突出表达，准备面试追问 |
| 待优化表达 | 经历真实存在，但缺少技术动作、结果、规模、职责边界或岗位关键词 | 改写 bullet，补充细节 |
| 真实能力缺口 | 资料中没有证据，不能直接写入简历 | 转化为学习建议或补项目建议 |
| 面试风险 | 简历中出现但证据弱、范围过大、描述过满或用户可能讲不清 | 准备追问答案或弱化表述 |

## 评估方式

- 不用虚假百分比分数制造精确感
- 使用档位：高/中/低 或 1-5 档
- 每项建议必须可执行

## 常见问题与建议

| 问题类型 | 具体表现 | 优化建议 |
|----------|----------|----------|
| 缺少量化结果 | "优化了性能"无指标 | 补充具体数字或范围 |
| 技术描述空泛 | "使用 Redis"无细节 | 改为"使用 Redis 缓存热点数据，响应时间降低 60%" |
| 职责边界不清 | "参与项目开发" | 改为"负责 XX 模块的后端开发" |
| 证据不足 | 写了"熟悉 XX"但无项目支撑 | 准备追问答案，或弱化为"了解 XX" |
| 范围过大 | "独立负责整个系统" | 明确具体负责的模块 |

## 风险等级

| 风险等级 | 说明 | 处理方式 |
|----------|------|----------|
| 高 | 容易被追问穿透，无法自圆其说 | 删除或大幅弱化 |
| 中 | 可能被追问细节，需要准备 | 准备追问答案 |
| 低 | 基本可信，但可以更具体 | 补充细节即可 |
