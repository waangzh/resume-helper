# Resume Intake

## 流程

1. 读取用户提供的简历、项目文档、作品链接、证书照片和会话材料。
2. 识别输入类型：结构化简历、非结构化草稿、口述经历、证书照片。
3. 按输入类型选择处理策略，提取基本信息、教育背景、技能栈、项目经历、实习经历、竞赛/科研、获奖证书。
4. 为每条经历记录证据来源：文件名、段落摘要、链接或用户原句。
5. 按目标岗位标记相关度：高、中、低、待确认。
6. 输出 `CandidateProfile` 和 `uncertain_items`。

## 输入类型处理

| 输入类型 | 处理策略 |
|----------|----------|
| 结构化简历 | 直接提取字段 |
| 非结构化草稿 | 按萃取模板追问 + 结构化 |
| 口述经历 | 时间线重构 + STAR 细节补全 |
| 证书照片 | 图片识别工具 + 提取关键字段 |

## 证书照片处理

当用户上传证书照片时：

1. 使用 AutoClaw 内置图片识别工具识别证书内容：
   - 使用 autoglm-file-upload 上传本地图片获取公网 URL
   - 使用 autoglm-image-recognition 识别图片内容
2. 提取关键信息：证书名称、获奖人、等级/分数、时间、发证单位
3. 转换为结构化经历记录
4. 低置信度内容标记为”待人工确认”
5. 若图片识别工具不可用，采用交互式追问请用户口述

**支持识别的证书类型：**
- 获奖证书、竞赛证书
- 英语等级证书（四六级、雅思、托福）
- 计算机等级证书、职业资格证书
- 实习证明、志愿者证书

## 非结构化草稿处理

当用户提供凌乱的校园经历描述时：

1. 识别经历类型（社团、兼职、竞赛、项目等）
2. 提取或推断时间线
3. 补全角色和职责信息
4. 追问关键细节用于 STAR 改写
5. 输出带”待确认”标记的结构化记录

**最小追问原则：** 只追问 1-3 个影响简历质量的关键问题，用户未回答时继续生成草案。

## 输出字段

| 字段 | 说明 | 证据要求 |
|------|------|----------|
| basic_info | 姓名、电话、邮箱、城市等 | 材料原文 |
| education | 学校、专业、学历、时间、成绩 | 材料原文 |
| skills | 技术栈，区分已证实/待确认/建议补强 | 项目证据支撑 |
| experiences | 项目、实习、竞赛、科研、社团经历 | 材料原文 + 证据来源 |
| evidence | 每条经历对应的来源片段 | 文件名/链接/原句/照片 |
| uncertain_items | 无法确认但影响质量的信息 | 标记待追问 |

## 解析原则

- 原文有歧义时保留原句，不擅自补全
- 联系方式、学校、公司、时间、数字成果必须来自材料
- 对”熟悉/了解/掌握”等技能词，结合项目证据判断可信度
- 证书照片识别低置信度内容标记”待确认”，不编造

## 证据可信度

| 级别 | 说明 | 处理方式 |
|------|------|----------|
| confirmed | 材料中明确出现 | 直接用于简历 |
| derived | 可由材料合理归纳 | 保持保守表达 |
| uncertain | 影响质量但无法确认 | 追问或标记待确认 |
| suggested | 只是建议补强 | 不能写成已具备能力 |
